# Azure VM Snapshot and Restore Guide

This guide explains how development team members can take and restore snapshots of Azure Virtual Machine (VM) OS disks for testing, development, and rollback scenarios.  

It's based on a page generated by ChatGPT.  The format isn't great, and the commands have been in a form that don't seem to work if used directly.    
I've converted the main one into some commands you can use in powershell.  They don't use the powershell azure interface though, but instead the az cli.  I think the powershell azure interface needs a module install to work.  

---

## Prerequisites

### RBAC Permissions

You must have one of the following roles 

 **at the Resource Group level**:

| Role | Purpose |
| --- | --- |
| Disk Snapshot Contributor | Allows creating snapshots of disks |
| Contributor | Allows create/restore operations, including replacing disks |

If you do not have these permissions, contact the platform/cloud administrators.

### VM Requirements

- VM must use 

 **Managed Disks**
- ### VM must not use Ephemeral OS Disks

### CLI Requirements

You must install the Azure CLI  

**https://learn.microsoft.com/en\-us/cli/azure/install\-azure\-cli?view\=azure\-cli\-latest**

## Taking a Snapshot (Azure Portal)

1. Open the 

 **Azure Portal**.
2. Navigate to 

 **Virtual Machines** and select the VM.
3. In the left menu, select 

 **Disks**.
4. Click the 

 **OS Disk** (typically named 

 `vmname_OsDisk_...`).
5. Click 

 **Create Snapshot**.
6. Enter a name following this convention:

```
         

         <vmname>-snapshot-<YYYYMMDD>


```
Example: 

 `webapp01-snapshot-20251110`
7. Select 

 **Standard HDD** as the storage type (cost\-efficient).
8. Click 

 **Review \+ Create**.

The snapshot will appear under 

 **Snapshots** in the Resource Group.

---

## Taking a Snapshot (CLI)

```
   

   

      $ResourceGroup = "MyResourceGroup"       # Replace with your RG name

      $DiskName = "myvm_OsDisk_1_abcdef"       # Replace with your VM's OS disk name

      $SnapshotName = "$($DiskName)-snapshot-$(Get-Date -Format 'yyyyMMdd-HHmm')"  


    

      az snapshot create  --resource-group $ResourceGroup  --name $SnapshotName  --source $DiskName

 
```

````
If you don't know the disk name: 

      

         az vm show -g MyResourceGroup -n MyVM --query "storageProfile.osDisk.name" -o tsv



   
```

## Restoring a Snapshot

Restoring a snapshot replaces the current OS disk on the VM.

> Important: This discards 

>  all changes made since the snapshot. Make sure the user understands this.

### Steps

1. Go to 

 Snapshots, select the snapshot.
2. Click 

 Create Disk.

	- Use a name such as: 

	 vmname-osrestored-YYYYMMDD
3. Go back to the VM → 

 Stop (Deallocate) the VM.
4. Go to 

 VM → Disks → Swap OS Disk.
5. Select the disk created in Step 2.
6. Start the VM.

## Restoring via CLI

To list snapshots:  

az snapshot list --resource-group $ResourceGroup --output table  

```
# Create disk from snapshot

az disk create `

  --resource-group $ResourceGroup `

  --name restoredDisk `

  --source $SnapshotName



# Stop VM

az vm deallocate -g $RESOURCE_GROUP -n $VM_NAME



# Replace OS disk

az vm update \

  -g $RESOURCE_GROUP \

  -n $VM_NAME \

  --os-disk restoredDisk



# Start VM

az vm start -g $RESOURCE_GROUP -n $VM_NAME


```

## Naming Conventions

| Resource | Format | Example |
| --- | --- | --- |
| Snapshot | <vm>-snapshot-<date> | devapp01-snapshot-20251110 |
| Restored Disk | <vm>-restored-<date> | devapp01-restored-20251110 |

## Good Practices

- Always snapshot before major system updates or configuration changes.
- Delete old snapshots periodically to control storage costs.
- For frequent rollback workflows, consider using 

 Shared Image Gallery instead.

## Troubleshooting

| Issue | Resolution |
| --- | --- |
| Can't create snapshot | Check RBAC role (Disk Snapshot Contributor required) |
| Restore option missing | Ensure VM is 
  stopped/deallocated first |
| Snapshot storage cost too high | Use 
  Standard HDD tier |
